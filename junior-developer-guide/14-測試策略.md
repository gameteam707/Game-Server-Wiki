# 14 — 測試策略

## 為什麼要寫測試？

> 「沒有測試的程式碼，就是無法驗證正確性的程式碼。」

測試不只是為了抓 Bug，更是為了：
- **安全重構**：改了程式碼，跑測試就知道有沒有弄壞東西
- **文件功能**：好的測試就是「可執行的規格書」
- **加速開發**：手動測試一個 API 要 30 秒，自動測試 0.1 秒

## Go 的測試慣例

```
規則：
1. 測試檔案以 _test.go 結尾
2. 測試函式以 Test 開頭
3. 參數是 *testing.T
4. 放在跟被測程式碼同一個目錄

範例：
usecase/auth/
├── uc.go          ← 被測程式碼
└── uc_test.go     ← 測試程式碼
```

## 測試金字塔

```
        ╱╲
       ╱  ╲         E2E Tests（端到端測試）
      ╱    ╲        少量：完整 API 流程
     ╱──────╲
    ╱        ╲       Integration Tests（整合測試）
   ╱          ╲      中量：UseCase + 真實 DB
  ╱────────────╲
 ╱              ╲     Unit Tests（單元測試）
╱                ╲    大量：純邏輯、Mock 依賴
╱──────────────────╲
```

| 層級 | 速度 | 可靠度 | 維護成本 | 適用範圍 |
|------|------|--------|---------|---------|
| Unit | 毫秒 | 高 | 低 | Domain、UseCase 邏輯 |
| Integration | 秒 | 中 | 中 | Repository、API 呼叫 |
| E2E | 分鐘 | 低 | 高 | 完整使用者場景 |

## 本專案的測試模式

### 1. UseCase 單元測試（Mock 依賴）

```go
// 檔案：admin-backend/server/internal/usecase/auth/uc_test.go

// ====== Step 1: 建立 Mock ======
type mockAdminUserRepo struct {
    user *entity.AdminUser
    err  error
}

// 實作 ports.AdminUserRepository 介面
func (m *mockAdminUserRepo) FindByUsername(_ context.Context, _ string) (*entity.AdminUser, error) {
    return m.user, m.err
}

func (m *mockAdminUserRepo) Create(_ context.Context, _ *entity.AdminUser) error {
    return nil
}

type mockLoginLimiter struct {
    blocked bool
}

func (m *mockLoginLimiter) IsBlocked(_ context.Context, _ string) bool {
    return m.blocked
}

func (m *mockLoginLimiter) RecordFailure(_ context.Context, _ string) {}
func (m *mockLoginLimiter) ClearFailures(_ context.Context, _ string) {}

// ====== Step 2: 撰寫測試 ======
func TestLogin_Success(t *testing.T) {
    // Arrange（準備）
    hashed, _ := bcrypt.GenerateFromPassword([]byte("admin123"), bcrypt.DefaultCost)
    repo := &mockAdminUserRepo{
        user: &entity.AdminUser{
            ID:             "user_001",
            Username:       "admin",
            HashedPassword: string(hashed),
            Role:           entity.RoleSuperAdmin,
        },
    }
    limiter := &mockLoginLimiter{blocked: false}

    uc := NewLoginUseCase(repo, limiter, "test-secret", "24h")

    // Act（執行）
    result, err := uc.Execute(context.Background(), &LoginCommand{
        Username: "admin",
        Password: "admin123",
        IP:       "127.0.0.1",
    })

    // Assert（驗證）
    if err != nil {
        t.Fatalf("expected success, got error: %v", err)
    }
    if result.Token == "" {
        t.Fatal("expected token to be non-empty")
    }
}

func TestLogin_WrongPassword(t *testing.T) {
    hashed, _ := bcrypt.GenerateFromPassword([]byte("admin123"), bcrypt.DefaultCost)
    repo := &mockAdminUserRepo{
        user: &entity.AdminUser{
            HashedPassword: string(hashed),
        },
    }
    limiter := &mockLoginLimiter{blocked: false}

    uc := NewLoginUseCase(repo, limiter, "test-secret", "24h")

    _, err := uc.Execute(context.Background(), &LoginCommand{
        Username: "admin",
        Password: "wrong-password",
        IP:       "127.0.0.1",
    })

    if err == nil {
        t.Fatal("expected error for wrong password")
    }
}

func TestLogin_Blocked(t *testing.T) {
    repo := &mockAdminUserRepo{}
    limiter := &mockLoginLimiter{blocked: true}  // 被封鎖

    uc := NewLoginUseCase(repo, limiter, "test-secret", "24h")

    _, err := uc.Execute(context.Background(), &LoginCommand{
        Username: "admin",
        Password: "admin123",
        IP:       "127.0.0.1",
    })

    if err == nil {
        t.Fatal("expected error for blocked IP")
    }
}
```

**觀察重點**：
- 因為 UseCase 依賴的是介面，所以可以輕鬆注入 Mock
- 每個測試案例測試一個特定情境
- 不需要資料庫、不需要 HTTP Server

### 2. SSE Hub 並行測試

```go
// 檔案：config-service/internal/infrastructure/sse/hub_test.go

func TestHub_BroadcastToClient(t *testing.T) {
    hub := NewHub()
    go hub.Run()  // Hub 在背景執行

    // 註冊一個客戶端
    ch := make(chan Event, 10)
    hub.Register(ch)

    // 廣播事件
    hub.Broadcast(Event{
        Type: "integrator_updated",
        Data: `{"id": "int_001", "name": "TestCo"}`,
    })

    // 驗證客戶端收到事件
    select {
    case evt := <-ch:
        if evt.Type != "integrator_updated" {
            t.Errorf("expected type integrator_updated, got %s", evt.Type)
        }
    case <-time.After(time.Second):
        t.Fatal("timeout waiting for event")
    }
}

func TestHub_MultipleClients(t *testing.T) {
    hub := NewHub()
    go hub.Run()

    // 註冊多個客戶端
    clients := make([]chan Event, 3)
    for i := range clients {
        clients[i] = make(chan Event, 10)
        hub.Register(clients[i])
    }

    // 廣播
    hub.Broadcast(Event{Type: "test", Data: "hello"})

    // 每個客戶端都應該收到
    for i, ch := range clients {
        select {
        case evt := <-ch:
            if evt.Data != "hello" {
                t.Errorf("client %d: expected hello, got %s", i, evt.Data)
            }
        case <-time.After(time.Second):
            t.Fatalf("client %d: timeout", i)
        }
    }
}
```

### 3. Table-Driven Tests（表格驅動測試）

Go 社群推崇的測試模式——用表格列出所有測試案例：

```go
func TestGameStatus_IsValid(t *testing.T) {
    tests := []struct {
        name   string
        status entity.GameStatus
        want   bool
    }{
        {"active is valid", entity.GameStatusActive, true},
        {"inactive is valid", entity.GameStatusInactive, true},
        {"empty is invalid", entity.GameStatus(""), false},
        {"unknown is invalid", entity.GameStatus("unknown"), false},
    }

    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            got := tt.status.IsValid()
            if got != tt.want {
                t.Errorf("GameStatus(%q).IsValid() = %v, want %v",
                    tt.status, got, tt.want)
            }
        })
    }
}
```

**好處**：新增測試案例只需要加一行到表格，不需要寫新的函式。

## Mock 的設計原則

### 在 Go 中，通常手寫 Mock 而非使用框架

```go
// 手寫 Mock：簡單、可控
type mockGameRepo struct {
    games  []*entity.Game
    err    error
    called bool           // 可以追蹤方法是否被呼叫
}

func (m *mockGameRepo) Create(_ context.Context, game *entity.Game) error {
    m.called = true
    if m.err != nil {
        return m.err
    }
    m.games = append(m.games, game)
    return nil
}
```

**為什麼不用 Mock 框架（如 gomock、mockery）？**
- Go 的介面通常方法很少，手寫 Mock 很快
- 手寫 Mock 更容易閱讀和除錯
- 不需要額外的程式碼生成步驟

## 執行測試

```bash
# 執行所有測試
go test ./...

# 執行特定套件的測試
go test ./internal/usecase/auth/...

# 顯示詳細輸出
go test -v ./internal/usecase/auth/...

# 執行特定測試函式
go test -run TestLogin_Success ./internal/usecase/auth/...

# 查看測試覆蓋率
go test -cover ./...

# 產生覆蓋率報告
go test -coverprofile=coverage.out ./...
go tool cover -html=coverage.out
```

## 給 Junior 的學習重點

### 你需要理解的概念

1. **AAA 模式**：Arrange（準備）→ Act（執行）→ Assert（驗證）
2. **Table-Driven Tests**：Go 的標準測試模式
3. **Mock vs Stub vs Fake**：不同的測試替身
4. **測試覆蓋率**：不是 100% 才好，但關鍵路徑一定要覆蓋

### 動手練習

1. 跑一次 `go test ./...`，看哪些服務有測試
2. 閱讀 `auth/uc_test.go`，理解每個測試案例在測什麼
3. 為現有的 UseCase 新增一個測試案例（例如：使用者不存在的情況）
4. 嘗試寫一個 Table-Driven Test
