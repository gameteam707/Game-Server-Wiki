# 13 — 部署與容器化

## Docker：為什麼需要容器化？

**問題**：「在我的電腦上可以跑啊！」

不同環境（開發機、測試機、正式機）的 Go 版本、系統函式庫、環境設定都可能不同。Docker 解決這個問題——**把應用和它的所有依賴打包成一個標準化的容器**。

## 多階段建置（Multi-stage Build）

本專案所有服務都使用多階段建置，這是 Docker 的最佳實踐：

```dockerfile
# 檔案：admin-backend/server/docker/Dockerfile

# ====== 第一階段：建置 ======
FROM golang:1.25-alpine AS builder
RUN apk --no-cache add git
WORKDIR /build

# 先複製依賴檔案（利用 Docker 快取層）
COPY go.mod go.sum ./
RUN go mod download

# 再複製原始碼
COPY . .

# 編譯成靜態連結的二進位檔
RUN CGO_ENABLED=0 GOOS=linux go build -o /server ./cmd/server/

# ====== 第二階段：執行 ======
FROM alpine:3.19
RUN apk --no-cache add ca-certificates
WORKDIR /app
COPY --from=builder /server .
RUN mkdir -p uploads
EXPOSE 8090
CMD ["./server"]
```

### 為什麼要分兩階段？

| | 只用一階段 | 多階段建置 |
|--|----------|-----------|
| 映像大小 | ~1GB（包含整個 Go SDK） | ~20MB（只有 Alpine + 二進位檔） |
| 安全性 | 映像中包含編譯工具鏈 | 只有執行所需的最少元件 |
| 啟動速度 | 慢（大映像需要下載） | 快（映像小） |

### 關鍵參數解釋

```
CGO_ENABLED=0   → 禁用 CGO，產生完全靜態連結的二進位
GOOS=linux      → 交叉編譯為 Linux（即使在 Mac 上建置）
ca-certificates → HTTPS 連線需要的根憑證
```

### Kubernetes 核心概念

```yaml
# 範例：gameservice 的 Deployment

apiVersion: apps/v1
kind: Deployment
metadata:
  name: gameservice
spec:
  replicas: 2                    # 跑 2 個副本
  selector:
    matchLabels:
      app: gameservice
  template:
    spec:
      containers:
      - name: gameservice
        image: gameservice:latest
        ports:
        - containerPort: 50051   # gRPC
        - containerPort: 9090    # Metrics
        env:
        - name: GRPC_PORT
          value: "50051"
        - name: MONGO_URI
          valueFrom:
            secretKeyRef:        # 機敏資料從 Secret 取
              name: mongo-secret
              key: uri
        resources:
          requests:
            cpu: 100m
            memory: 128Mi
          limits:
            cpu: 500m
            memory: 512Mi
```

### K8s 概念速查

| 概念 | 說明 | 類比 |
|------|------|------|
| **Pod** | 最小部署單位，包含一或多個容器 | 一台虛擬機 |
| **Deployment** | 管理 Pod 的副本數量和更新策略 | 自動管理的 Pod 群組 |
| **Service** | Pod 的穩定網路入口 | 負載均衡器 |
| **ConfigMap** | 非機敏設定資料 | 環境變數檔案 |
| **Secret** | 機敏設定資料（密碼、Token） | 加密的環境變數 |
| **StatefulSet** | 有狀態服務（如 MongoDB） | 需要固定儲存的 Pod |
| **Ingress** | 外部流量入口 | 反向代理（Nginx） |
| **Kustomize** | K8s 原生的設定管理工具 | 環境差異管理 |
| **KEDA** | 根據事件自動擴縮 | 流量多就多開 Pod |

### Kustomize 的作用

```
問題：staging 和 production 的設定不同（副本數、映像版本、資源限制）

解法：
base/     → 定義共用的基礎清單
overlays/ → 每個環境覆蓋不同的值

# 部署到 staging
kubectl apply -k overlays/staging/

# 部署到 GKE
kubectl apply -k overlays/gke-staging/
```
