# 11 — 微服務通訊

## 為什麼微服務需要通訊？

單體應用中，模組之間可以直接函式呼叫。微服務架構中，服務跑在不同的程序（甚至不同的機器），必須透過 **網路協定** 溝通。

本專案使用了三種通訊模式：

```
┌──────────────────────────────────────────────────────┐
│                                                      │
│   同步通訊（請求-回應）                                  │
│   ├── HTTP (REST)  ← admin-backend, gameapi          │
│   └── gRPC         ← gameservice                     │
│                                                      │
│   非同步通訊（事件驅動）                                  │
│   ├── SSE          ← config-service → gameservice    │
│   └── Kafka        ← gameservice → 下游消費者          │
│                                                      │
└──────────────────────────────────────────────────────┘
```

## 1. gRPC — 服務間的高效通訊

### 什麼是 gRPC？

gRPC 是 Google 開發的 RPC（Remote Procedure Call）框架。簡單說，它讓你 **像呼叫本地函式一樣呼叫遠端服務**。

### 與 HTTP REST 的比較

| 特性 | HTTP REST | gRPC |
|------|-----------|------|
| 傳輸格式 | JSON（文字） | Protocol Buffers（二進位） |
| 效能 | 較慢（序列化/反序列化 JSON） | 較快（二進位壓縮） |
| 型別安全 | 弱（JSON 是動態的） | 強（.proto 定義嚴格型別） |
| 適用場景 | 對外 API、瀏覽器直接呼叫 | 服務間內部通訊 |
| 生態工具 | Postman、curl | grpcurl、BloomRPC |

### Protocol Buffers 定義

```protobuf
// 檔案：sfc-stream-game/proto/game.proto

syntax = "proto3";

package gamepb;

service GameService {
    rpc InitGame(InitGameRequest) returns (InitGameResponse);
    rpc Spin(SpinRequest) returns (SpinResponse);
    rpc GetBalance(GetBalanceRequest) returns (GetBalanceResponse);
}

message SpinRequest {
    string session_id = 1;
    double bet_amount = 2;
    string currency = 3;
}

message SpinResponse {
    double win_amount = 1;
    repeated Symbol symbols = 2;
    // ...
}
```

`.proto` 檔案會用工具（`protoc`）自動生成 Go 程式碼，不需要手寫。

### gRPC Server 端（gameservice）

```go
// 檔案：gameservice/internal/adapter/in/grpc/server.go

type GameServer struct {
    gamepb.UnimplementedGameServiceServer  // 嵌入預設實作
    spinUC spin.UseCase
}

func (s *GameServer) Spin(ctx context.Context, req *gamepb.SpinRequest) (*gamepb.SpinResponse, error) {
    // 1. Protobuf Request → UseCase Command
    cmd := &spin.Command{
        SessionID: req.SessionId,
        BetAmount: req.BetAmount,
    }

    // 2. 執行業務邏輯
    result, err := s.spinUC.Execute(ctx, cmd)
    if err != nil {
        return nil, status.Errorf(codes.Internal, "spin failed: %v", err)
    }

    // 3. UseCase Result → Protobuf Response
    return &gamepb.SpinResponse{
        WinAmount: result.WinAmount,
    }, nil
}
```

### gRPC Client 端（gameapi 呼叫 gameservice）

```go
// gameapi 透過 gRPC 呼叫 gameservice
```

## 2. SSE — 即時設定推播

### 什麼是 SSE？

SSE（Server-Sent Events）是一種 **伺服器主動推送資料給客戶端** 的技術。與 WebSocket 不同，SSE 是 **單向** 的（伺服器 → 客戶端）。

### 為什麼用 SSE？

```
情境：管理員在 admin-backend 修改了遊戲 RTP

不用 SSE（輪詢模式）：
  gameservice 每 5 秒問一次：「設定有更新嗎？」「沒有。」
  gameservice 每 5 秒問一次：「設定有更新嗎？」「沒有。」
  gameservice 每 5 秒問一次：「設定有更新嗎？」「有！」
  → 浪費大量無效請求，而且延遲最多 5 秒

用 SSE（推播模式）：
  gameservice 建立一條 SSE 連線，然後等著
  ...（沒有無效請求）...
  config-service 收到變更，立即推送：「遊戲 RTP 更新了！」
  → 零浪費，即時通知
```

### SSE Hub 實作

```go
// 檔案：config-service/internal/infrastructure/sse/hub.go

type Event struct {
    Type string
    Data string
}

type Hub struct {
    clients    map[chan Event]bool  // 已連線的客戶端
    register   chan chan Event      // 新連線
    unregister chan chan Event      // 斷線
    broadcast  chan Event           // 要廣播的事件
}

func (h *Hub) Run() {
    for {
        select {
        case client := <-h.register:
            h.clients[client] = true

        case client := <-h.unregister:
            delete(h.clients, client)
            close(client)

        case event := <-h.broadcast:
            // 廣播給所有已連線的客戶端
            for client := range h.clients {
                select {
                case client <- event:
                default:
                    // 客戶端來不及消費，跳過
                    close(client)
                    delete(h.clients, client)
                }
            }
        }
    }
}
```

## 3. Kafka — 事件驅動架構

### 什麼是 Kafka？

Apache Kafka 是一個分散式訊息串流平台。可以把它想像成一個 **超高效能的訊息佇列**。

### 為什麼需要 Kafka？

```
情境：玩家下注，gameservice 需要做很多事

同步做法（❌ 慢）：
  玩家下注 → 扣款 → 計算結果 → 入款 → 寫下注紀錄 → 發送統計 → 回傳結果
  （玩家要等所有步驟完成才能看到結果）

用 Kafka（✅ 快）：
  玩家下注 → 扣款 → 計算結果 → 入款 → 回傳結果（玩家看到結果了！）
                                          ↓
                                   發送 Kafka Event
                                          ↓
                              ┌───────────┴───────────┐
                              ▼                       ▼
                         寫下注紀錄              發送統計報表
                      （非同步處理）            （非同步處理）
```

### Kafka Event Publisher

```go
// 檔案：gameservice/internal/adapter/out/messaging/kafka_publisher.go

type KafkaPublisher struct {
    writer *kafka.Writer
    topic  string
}

func (p *KafkaPublisher) PublishBetEvent(ctx context.Context, event BetEvent) error {
    data, _ := json.Marshal(event)

    return p.writer.WriteMessages(ctx, kafka.Message{
        Key:   []byte(event.SessionID),
        Value: data,
    })
}
```

## 4. Outbox Pattern — 可靠的事件發布

### 問題：如果 Kafka 發送失敗怎麼辦？

```
❌ 危險的做法：
  1. 存下注紀錄到 MongoDB    ← 成功
  2. 發送事件到 Kafka         ← 失敗！
  → 資料庫有紀錄，但事件沒發出去，下游系統不知道有這筆下注
```

### Outbox Pattern 的解法

```
✅ Outbox Pattern：
  1. 在同一個 MongoDB Transaction 中：
     a. 存下注紀錄
     b. 存一筆 Outbox 訊息（pending 狀態）
  2. 背景 Worker 定期掃描 Outbox：
     a. 找到 pending 的訊息
     b. 發送到 Kafka
     c. 標記為 sent
  3. 如果發送失敗：
     a. 重試
     b. 超過重試次數 → 進入 Dead Letter Queue
     c. 發送 Webhook 告警
```

```go
// gameservice 中的 Outbox Worker（概念）

func (w *OutboxWorker) Poll(ctx context.Context) {
    ticker := time.NewTicker(w.pollInterval)
    for range ticker.C {
        messages, _ := w.outboxRepo.FindPending(ctx)
        for _, msg := range messages {
            if err := w.kafkaPublisher.Publish(ctx, msg); err != nil {
                w.outboxRepo.IncrementRetry(ctx, msg.ID)
                if msg.RetryCount >= w.maxRetries {
                    w.outboxRepo.MoveToDeadLetter(ctx, msg.ID)
                    w.webhookAlert(msg)
                }
                continue
            }
            w.outboxRepo.MarkSent(ctx, msg.ID)
        }
    }
}
```

## 通訊模式選擇指南

| 場景 | 建議模式 | 原因 |
|------|---------|------|
| 需要立即回應 | HTTP / gRPC | 同步等待結果 |
| 服務間高頻通訊 | gRPC | 二進位傳輸效率高 |
| 對外 API / 瀏覽器 | HTTP REST | 通用性最好 |
| 即時設定同步 | SSE | 單向推播、連線成本低 |
| 不需要立即處理 | Kafka | 非同步解耦、削峰 |
| 需要可靠投遞 | Kafka + Outbox | 確保訊息不丟失 |
| 即時雙向通訊 | WebSocket | 玩家遊戲畫面（connector 使用）|

## 給 Junior 的學習重點

### 你需要理解的概念

1. **同步 vs 非同步通訊**：同步等回應、非同步不等
2. **Protocol Buffers**：Google 的二進位序列化格式
3. **Server-Sent Events**：HTTP 長連線的推播機制
4. **Publish/Subscribe 模式**：發布者不知道誰在訂閱
5. **Outbox Pattern**：分散式系統中確保資料一致性的經典做法
