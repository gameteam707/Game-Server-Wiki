# 03 — 目錄結構與分層對照

## 總體原則

本專案所有 Go 服務都遵循相同的目錄結構慣例，一旦你理解了一個服務的結構，其他服務也能快速上手。

## 標準目錄結構

```
service-name/
├── cmd/
│   └── server/
│       └── main.go              ← 程式進入點，組裝所有元件
├── internal/                    ← Go 的可見性控制，外部 module 無法 import
│   ├── domain/                  ← 【第一層】核心領域
│   │   └── entity/              # 領域實體，零外部依賴
│   ├── usecase/                 ← 【第二層】業務邏輯
│   │   └── {feature}/           # 以功能命名的子目錄
│   │       ├── contract.go      # UseCase 介面 + Command/Result 結構
│   │       ├── uc.go            # UseCase 實作
│   │       └── ports/           # 這個 UseCase 需要的外部依賴介面
│   ├── adapter/                 ← 【第三層】轉接層
│   │   ├── in/                  # 入站（外部 → 系統）
│   │   │   ├── http/            # HTTP Handler
│   │   │   │   ├── *_handler.go
│   │   │   │   ├── middleware/
│   │   │   │   └── dto/
│   │   │   │       ├── request/
│   │   │   │       └── response/
│   │   │   └── grpc/            # gRPC Handler（gameservice 使用）
│   │   ├── out/                 # 出站（系統 → 外部）
│   │   │   ├── repository/      # 資料庫操作
│   │   │   │   └── mongo/
│   │   │   ├── gateway/         # 外部 API 呼叫
│   │   │   └── messaging/       # 訊息佇列（Kafka）
│   │   └── presenter/           # 統一回應格式
│   └── infrastructure/          ← 【第四層】基礎設施
│       ├── config/              # 設定檔讀取
│       ├── di/                  # 依賴注入容器
│       ├── mongodb/             # MongoDB 連線
│       ├── redis/               # Redis 連線
│       └── server/              # 路由設定
├── docker/
│   └── Dockerfile               # 多階段建置
├── go.mod
└── go.sum
```

## in / out 的含義

這是 Hexagonal Architecture（六邊形架構）的命名慣例：

```
         ┌──────────────────────────┐
         │                          │
  HTTP ──▶   in/http/handler.go     │
         │         │                │
  gRPC ──▶   in/grpc/server.go      │
         │         │                │
         │    UseCase Layer         │
         │         │                │
         │  out/repository ────────▶── MongoDB
         │  out/gateway ───────────▶── 外部 API
         │  out/messaging ─────────▶── Kafka
         │                          │
         └──────────────────────────┘
```

- **in**（入站）：外部世界呼叫我們的系統（HTTP Request、gRPC Call）
- **out**（出站）：我們的系統呼叫外部世界（存資料庫、呼叫 API、發送訊息）

## `internal/` 的意義

Go 語言有一個特殊慣例：`internal/` 目錄下的程式碼 **只能被同一個 module 內的程式碼 import**。

這不只是慣例，是 Go 編譯器強制執行的規則。這代表其他服務無法直接 import admin-backend 的 internal 程式碼，確保服務之間的邊界清晰。

## 給 Junior 的練習建議

1. 選一個你比較熟悉的功能（例如「幣別管理」），從 handler → usecase → repository 完整追一遍程式碼
2. 畫出那個功能的呼叫鏈（哪個 struct 呼叫哪個 method）
3. 找出 UseCase 定義了哪些 Port（介面），以及對應的 Adapter 實作在哪裡
