# 07 — 遊戲類型擴展指南

## 新增遊戲的決策樹

當需要新增一款遊戲時，按以下步驟決定架構方向：

```
Step 1: 結果來源
├── RNG（由 math-lib 決定）
│   └── Step 2a: 回合模式
│       ├── 每人獨立一局 → 【單人 PvE Spin 模式】
│       │   參考：02-單人遊戲流程-Spin-Lifecycle.md
│       │   範例：CashMachine, XiaoMali
│       │
│       └── 多人共享一局 → 【多人 RNG 共享局】
│           需擴展：Round 管理 + 共享 RNG 結果
│           （目前無現成範例，需基於 Round Lifecycle 擴展）
│
└── 外部信號（真實賽事/第三方 API）
    └── Step 2b: 投注模式
        ├── 選邊投注（Parimutuel）→ 【多人共享回合模式】
        │   參考：03-多人遊戲流程-Round-Lifecycle.md
        │   範例：Kung Fu Chicken
        │
        └── 固定賠率投注 → 【多人固定賠率模式】
            需擴展：後台設定賠率 + 固定 payout 計算
```

---

## 按遊戲類型的架構選型

### 類型 A：單人 PvE Spin

| 項目 | 設計 |
|------|------|
| 結果引擎 | math-lib `GameEngine` 介面 |
| gRPC Service | 複用 `GameService.Spin` |
| WS Route | `game.spin` |
| 狀態機 | BetRecord 個人狀態 |
| Redis | SpinState 暫存 |
| 新增工作量 | math-lib 新增 Engine + gameservice 註冊 |

**新增一款 Spin 遊戲只需要：**
1. 在 math-lib 實作 `GameEngine` 介面
2. 在 `engine/registry.go` 註冊
3. Config Service 加入 GameConfig
4. 無需改動 gameservice / connector

### 類型 B：多人共享回合（Parimutuel）

| 項目 | 設計 |
|------|------|
| 結果引擎 | 外部信號 |
| gRPC Service | 獨立 Service（如 `KungFuChickenService`） |
| WS Route | 遊戲前綴路由（如 `kfc.*`） |
| 狀態機 | Round 全局 + Bet 個人 |
| Redis | 投注池計數器 |
| 新增工作量 | 中等（UseCase + Adapter + Admin） |

**新增一款共享回合遊戲需要：**
1. 定義投注選項（BetOption）
2. 定義 Round Entity（可能可以泛化複用功夫雞的）
3. 定義 Bet Entity
4. 實作 UseCase（PlaceBet / Settle / RoundMgmt）
5. 新增 Proto 或複用 KFC Proto（若遊戲結構相似）
6. 新增 WS Routes + Handler
7. 新增 Admin API

## 擴展點清單

### Domain 層

| 擴展點 | 何時需要 | 如何擴展 |
|--------|---------|---------|
| 新增 Entity | 新遊戲有獨特的資料結構 | `domain/entity/` 新增檔案 |
| 新增 Value Object | 新的狀態/選項類型 | `domain/entity/` 新增檔案 |
| 新增 Domain Port | 新的外部依賴介面 | `domain/port/` 新增介面 |
| 複用 BetStatus | 所有遊戲的投注狀態 | 直接 import，不需修改 |

### UseCase 層

| 擴展點 | 何時需要 | 如何擴展 |
|--------|---------|---------|
| 新增 UseCase | 新的業務流程 | `usecase/{game}_{feature}/` 新增目錄 |
| 新增 Ports | UseCase 需要新的依賴 | `usecase/{feature}/ports/` 新增介面 |
| 複用賠率引擎 | 其他 Parimutuel 遊戲 | 直接呼叫 `kfc_odds.Calculate()` |
| 複用 Outbox | 任何需要補償的場景 | `outbox.OutboxWriter.Write()` |

### Adapter 層

| 擴展點 | 何時需要 | 如何擴展 |
|--------|---------|---------|
| 新增 gRPC Service | 新遊戲類型 | `proto/` 新增 .proto + `adapter/in/grpc/` 新增 server |
| 新增 WS Route | 新遊戲需要即時通訊 | `codec/packet.go` 加常量 + `handler/` 新增 handler |
| 新增 Repository | 新的 Collection | `adapter/out/repository/mongo/` 新增 repo + document |
| 新增 Redis Store | 新的快取需求 | `adapter/out/redis/` 新增 store |
| 複用 WalletClient | 所有遊戲 | 直接注入，不需修改 |
| 複用 ConfigCache | 所有遊戲 | 直接注入，不需修改 |

### Infrastructure 層

| 擴展點 | 何時需要 | 如何擴展 |
|--------|---------|---------|
| DI Container | 新增 UseCase 或 Adapter | `di/container.go` 新增欄位和初始化 |
| Migration | 新的 Collection 或 Index | `mongodb/migration/migrations/` 新增 migration |
| gRPC Server | 新增 gRPC Service | `infrastructure/server/` 註冊新 service |

## Checklist：新增一款遊戲

### 前置準備

- [ ] 確定遊戲類型（參考決策樹）
- [ ] 確定投注選項和規則
- [ ] 確定結果來源（RNG / 外部）
- [ ] 確定賠率模型（固定 RTP / Parimutuel / 後台設定）
- [ ] 在 `Game-Server-Wiki/game-flow-guide/` 撰寫設計文件
- [ ] 在 `gameservice/prd/` 撰寫 PRD

### Phase 1：Domain

- [ ] 定義 Entity（`domain/entity/`）
- [ ] 定義 Value Objects（狀態、選項等）
- [ ] 定義 Domain Events
- [ ] 定義 Global Ports（如有新的外部依賴）
- [ ] `go build` 通過

### Phase 2：UseCase

- [ ] 定義 Contract（UseCase 介面 + Command/Result）
- [ ] 定義 UseCase-level Ports
- [ ] 實作 UseCase
- [ ] 撰寫 UseCase 單元測試
- [ ] 測試覆蓋率 > 90%

### Phase 3：Adapter

- [ ] 定義 Proto（如需新 gRPC Service）
- [ ] 編譯 Proto（`protoc`）
- [ ] 實作 gRPC Server
- [ ] 實作 MongoDB Repository + Document 轉換
- [ ] 實作 Redis Store（如需要）
- [ ] 新增 MongoDB Migration（indexes）
- [ ] 擴展 DI Container
- [ ] `go build` 通過

### Phase 4：Connector

- [ ] 新增 WS Route 常量（`codec/packet.go`）
- [ ] 實作 WS Handler
- [ ] 實作 gRPC Client（呼叫 GameService）
- [ ] 註冊路由（`handler.go`）
- [ ] 擴展 DI Container
- [ ] `go build` 通過

### Phase 5：Admin Backend

- [ ] 實作管理 UseCase（如需後台管理）
- [ ] 實作 HTTP Handler + DTO
- [ ] 實作 gRPC Client
- [ ] 註冊路由 + 權限
- [ ] `go build` 通過

### Phase 6：Push 推播

- [ ] 確定推播事件類型和路由
- [ ] Kafka Topic 設定
- [ ] 確認 PushProducer 是否需要修改
- [ ] 測試推播鏈路

### Phase 7：整合測試

- [ ] 完整生命週期測試
- [ ] 異常場景測試（各種失敗 + 補償）
- [ ] 併發測試（如需多人）
- [ ] 使用 Special_Game_Pipeline 環境 E2E 測試

### Phase 8：部署

- [ ] kfc-k8s 新增/更新 Deployment 設定
- [ ] 環境變數設定
- [ ] Kafka Topic 建立
- [ ] MongoDB Collection 確認（Migration 自動建立）
- [ ] Grafana Dashboard 設定（如需獨立監控）

## 命名慣例

| 項目 | 慣例 | 範例 |
|------|------|------|
| 遊戲 ID | 全小寫英文 | `kungfuchicken`, `cashmachine` |
| WS Route 前綴 | 遊戲縮寫 + `.` | `kfc.`, `game.` |
| Kafka Topic | 遊戲縮寫 + `-` + 用途 | `kfc-bet-records`, `kfc-round-events` |
| MongoDB Collection | 遊戲縮寫 + `_` + 用途 | `kfc_rounds`, `kfc_bets` |
| UseCase 目錄 | 遊戲縮寫 + `_` + 功能 | `kfc_place_bet`, `kfc_settle` |
| Proto Package | 遊戲縮寫 + `pb` | `kfcpb` |
| gRPC Service | 遊戲全名 + `Service` | `KungFuChickenService` |
| Tracing Span | 遊戲縮寫 + `.` + 步驟 | `kfc.placeBet.debitWallet` |
