# 01 — 通用遊戲流程架構

## 遊戲生命週期的抽象模型

不論是單人 Spin 還是多人共享回合，所有遊戲都會經歷以下六個階段：

```
1. 連線認證     玩家透過 WebSocket 連線，Token 驗證身份
       ↓
2. 遊戲初始化   取得遊戲設定（賠率/規則/投注範圍）、查詢餘額
       ↓
3. 下注         扣款（Debit）→ 記錄投注
       ↓
4. 結算         決定結果（RNG / 外部信號）→ 計算輸贏
       ↓
5. 派彩         贏家入帳（Credit）→ 更新狀態
       ↓
6. 紀錄與事件   持久化投注紀錄、發布領域事件、更新統計
```

**差異在於各階段的「觸發方式」與「結果來源」不同**——單人遊戲由玩家驅動且即時完成，多人遊戲由管理員控制且分階段進行。

### 各服務在遊戲流程中的角色

| 服務 | 角色 | 遊戲流程中負責的階段 |
|------|------|---------------------|
| **Connector** | 玩家連線閘道 | 連線認證、WS 訊息路由、推播 |
| **GameService** | 遊戲核心邏輯 | 下注、結算、派彩 |
| **Platform (Wallet)** | 金流平台 | 扣款（Debit）、入帳（Credit）、取消（Cancel） |
| **Config Service** | 設定中心 | 遊戲初始化（提供設定） |
| **PushProducer** | 推播服務 | 即時通知（賠率更新、狀態變更） |
| **Kafka** | 事件匯流排 | 紀錄與事件（非同步持久化） |
| **MongoDB** | 持久化儲存 | 紀錄與事件（最終儲存） |
| **Admin Backend** | 管理介面 | 回合管理（多人遊戲）、遊戲設定 |

## BetStatus 狀態機

所有遊戲的投注紀錄都遵循相同的狀態流轉：

```
                    ┌──────────────┐
                    │   Confirmed  │  ← 扣款成功，投注已確認
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │    Settled   │  ← 結果已確定（RNG / 外部信號）
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              ▼                         ▼
    ┌─────────────────┐      ┌─────────────────┐
    │   Distributed   │      │    Refunded     │
    │  派彩完成         │      │  退款完成        │
    └─────────────────┘      └─────────────────┘
```

| 狀態 | 意義 | 觸發條件 |
|------|------|---------|
| **Confirmed** | 扣款成功，投注已記錄 | Wallet Debit 成功 |
| **Settled** | 結果已確定，知道輸贏 | RNG 計算完成 / 外部信號到達 |
| **Distributed** | 贏家已入帳 | Wallet Credit 成功（WinAmount > 0） |
| **Refunded** | 投注已退款 | 異常補償（Credit 失敗、回合取消等） |

> **程式碼位置**：`gameservice/internal/domain/entity/bet_status.go`

### 單人 vs 多人的狀態流轉差異

| 面向 | 單人 Spin | 多人 Round |
|------|----------|-----------|
| Confirmed → Settled | 毫秒級（同步 RNG） | 可能數分鐘（等待外部結果） |
| 結算範圍 | 一次結算一筆 | 一次批次結算整個回合所有投注 |
| Refunded 觸發 | Debit timeout / RNG failure | 回合取消 / 結算失敗補償 |

## 跨遊戲共用的基礎設施

以下元件已在 gameservice 中實作，新遊戲可直接複用：

### 1. Wallet Client（無縫錢包）

玩家資金由平台管理，遊戲商透過 HTTP 呼叫平台端點操作金流。

| 操作 | 方法 | 用途 |
|------|------|------|
| `Debit` | POST | 扣款（下注時） |
| `Credit` | POST | 入帳（派彩時） |
| `Cancel` | POST | 取消（補償退款） |
| `GetBalance` | POST | 查詢餘額 |

- **重試策略**：Exponential Backoff + Jitter（最多 5 次，基礎延遲 200ms，上限 5s）
- **冪等性**：透過 `TxID` 確保重複呼叫不重複扣款/入帳
- **程式碼位置**：`gameservice/internal/adapter/out/gateway/wallet_gateway.go`

### 2. Config Cache + SSE

記憶體快取遊戲設定，啟動時 Full Sync + SSE 即時增量更新。

| 設定類型 | 用途 |
|---------|------|
| `IntegratorConfig` | 平台商的 Wallet 端點 |
| `CurrencyConfig` | 匯率（BetAmountUSD 計算） |
| `GameConfig` | 遊戲狀態（active / maintenance） |
| `PlatformEndpointConfig` | 細粒度端點（per IID + GameType） |

- **程式碼位置**：`gameservice/internal/adapter/out/configcache/`

### 3. Event Publisher（Kafka 領域事件）

將業務事件發布到 Kafka `bet-events` topic，供下游消費（分析、報表、推播）。

```go
type DomainEvent struct {
    EventType   string      // "bet.confirmed", "bet.settled", "bet.completed", "bet.refunded"
    AggregateID string      // RoundID
    OccurredAt  time.Time
    Payload     interface{}
}
```

- **程式碼位置**：`gameservice/internal/adapter/out/messaging/kafka_event_publisher.go`

### 4. Outbox Pattern（補償機制）

當 Wallet 操作或 Kafka 發布失敗時，寫入 MongoDB `event_outbox`，由 Background Worker 重試。

```
操作失敗 → 寫入 Outbox (MongoDB) → Worker 定期掃描 → 重試 → 成功則標記 sent
                                                    → 超過重試次數 → 移入 Dead Letter
```

- **程式碼位置**：`gameservice/internal/adapter/out/outbox/`

### 5. SpinLock / ConcurrencyLock（分佈式鎖）

Redis 分佈式鎖，防止同一玩家併發操作（例如同時發起兩次 Spin）。

- **程式碼位置**：Connector 層的 Redis session lock

### 6. Observability（OTel Tracing）

所有關鍵步驟自動或手動建立 Span：

| 層級 | Tracing 方式 |
|------|-------------|
| HTTP 入站（Gin） | `ginmw.Trace()` 自動 |
| gRPC 入站/出站 | `grpcmw.ServerOption()` / `ClientOption()` 自動 |
| MongoDB / Redis | Driver 層自動 |
| UseCase 業務邏輯 | `tracing.Start(ctx, "useCase.step")` 手動 |

- **Span 命名慣例**：`{useCaseName}.{stepName}`，例 `spin_v2.debitWallet`

## 遊戲類型分類與選型指引

| 維度 | 單人 PvE (Spin) | 多人共享回合 (Round) |
|------|----------------|---------------------|
| **結果來源** | math-lib RNG | 外部信號 / 管理員輸入 |
| **回合歸屬** | 個人 | 共享 |
| **賠率模型** | 固定 RTP | 動態 Parimutuel / 後台設定 |
| **下注觸發** | 玩家驅動 | 管理員開盤 |
| **結算時機** | 即時（Spin 完成） | 延遲（等待結果） |
| **核心 Entity** | BetRecord | Round + Bet |
| **Wallet 操作** | 同步（Debit → RNG → Credit） | 非同步（Debit 在下注時，Credit 在結算時） |
| **推播需求** | 低（僅個人結果） | 高（賠率更新、狀態變更廣播） |

### 選型決策樹

```
新遊戲需求
    │
    ├── 結果由 RNG 決定？
    │     ├── 是 → 每人獨立一局？
    │     │         ├── 是 → 單人 PvE Spin（參考 02-單人遊戲流程）
    │     │         └── 否 → 多人 RNG 共享局（需擴展）
    │     │
    │     └── 否 → 結果來自外部？
    │               ├── 是 → 多人共享回合（參考 03-多人遊戲流程）
    │               └── 否 → 需另外設計
    │
    └── 詳見 07-遊戲類型擴展指南
```
